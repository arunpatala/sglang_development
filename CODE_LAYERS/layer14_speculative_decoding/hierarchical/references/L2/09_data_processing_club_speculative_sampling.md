## Summary

Speculative decoding has three key properties that make it remarkable:

1. **Exact** — does not alter the output distribution.
2. **Training-free** — applies directly to existing models.
3. **Easy to implement** — clean decoding loop.

> *"The limitation of sequential generation is not an unbreakable law of physics. It is an engineering hurdle that is being cleared with elegant theoretical insights."*

---

## Why decoding is slow

An autoregressive Transformer generates tokens sequentially: compute logits → sample → append → repeat. The strict dependence chain (token t+1 cannot start until token t is done) prevents full GPU parallelism even on the fastest hardware.

**Speculative decoding attacks this bottleneck:** use a cheap draft model to propose multiple tokens quickly, then use the target model's parallel computation to verify them all at once.

---

## The core loop

Fix γ (the number of draft tokens per round):

1. Draft model generates γ tokens: x₁, x₂, ..., xγ.
2. Target model **verifies these tokens in parallel** (one forward pass).
3. Accept/reject from left to right:
   - If x₁, ..., xᵢ accepted but xᵢ₊₁ rejected → confirm x₁, ..., xᵢ and the target's own xᵢ₊₁'.
   - If all γ accepted → confirm x₁, ..., xγ and the target's γ+1'th token.
4. Repeat from the new prefix.

## The acceptance criterion: SpecJudge and SpecNormalize

The naive approach (accept only exact matches) fails even when draft = target perfectly — with a 100-token vocabulary, the probability of exact match is 1%. You need a probabilistic rule.

### SpecJudge (the acceptance decision)

For each position i, given draft probability `q_d(xᵢ | x_{:i-1})` and target probability `p_t(xᵢ | x_{:i-1})`:

```
(1) If q_d(xᵢ) ≤ p_t(xᵢ): Accept always.

(2) Otherwise: Accept with probability p_t(xᵢ) / q_d(xᵢ).
    (Sample r ~ Uniform(0,1); accept if r < p_t / q_d)
```

### SpecNormalize (the correction on rejection)

When a token is rejected, sample a replacement from:

```
p_resid(x) = normalize( max(0, p_t(x) - q_d(x)) )
```

---

## Why this is exactly p_t — the decomposition proof

Define D = {x | q_d(x) > p_t(x)} (tokens where draft overestimates).

**Case 1: x ∈ D** (draft overestimates)
- Generated by draft with probability q_d(x)
- Accepted with probability p_t(x) / q_d(x) via SpecJudge
- Net probability: q_d(x) × (p_t(x) / q_d(x)) = **p_t(x)** ✓

**Case 2: x ∉ D** (target ≥ draft)
- Accepted whenever generated by draft: probability q_d(x)
- When a token in D is rejected (probability C = Σ_D (1 - p_t/q_d) dq_d), sample x from p_resid
- Net additional probability: C × (p_t(x) - q_d(x)) / C = p_t(x) - q_d(x)
- Total: q_d(x) + (p_t(x) - q_d(x)) = **p_t(x)** ✓

In both cases, the token is emitted with exactly the target probability.

> **Key difference from standard rejection sampling:** standard RS may reject repeatedly. SpecJudge guarantees resolution in at most one round — accept via (1), accept via (2), or reject and sample once from SpecNormalize. No re-sampling loop.

## Does it actually speed things up?

### Leviathan et al. (2023) results — T5-XXL (11B) + T5-Small (77M)

| Task | γ | Acceptance rate | Speedup |
|------|---|-----------------|---------|
| Translation | 5 | 53% | 2.3× |
| Summarization | 7 | 75% | 3.4× |

### The bigram result (the most surprising finding)

Using a **bigram model** (no GPU, negligible cost) as the draft:
- Acceptance rate: ~20%
- γ = 3
- Speedup: **1.25×**

> *"Even a 'dumb' draft model provides a speed boost because verification is parallel."*

## Extensions covered in this article

### 1. DistillSpec — align draft to target via knowledge distillation

Standard speculative decoding: draft and target are independent; acceptance depends on how well q_d matches p_t.

**Subtle trap:** higher token-level accuracy of the draft model doesn't always help. If the draft is *more confident* than the target about a token the target is uncertain about, that token will be rejected.

**DistillSpec fix:** distill from target → draft so q_d approximates p_t. Result: 10–45% additional speedup.

### 2. Online Speculative Decoding — fine-tune the draft on the fly

After verification, store mismatches in a training buffer and continuously fine-tune the draft on the target's outputs. Adapts to shifts in query distribution without pre-collecting training data. Especially effective for domain-specific deployment.

### 3. Draft & Verify — truncate target layers as draft

Remove some transformer layers from the target to create a faster draft. Leverages skip connections (xˡ⁺¹ = xˡ + f(xˡ)) — modern LLMs have small representation changes across layers, so removing layers doesn't degrade accuracy much. Uses Bayesian optimization to find optimal layer removal. Speedup: up to 1.99×. Benefit: single model deployment.

### 4. LayerSkip — intermediate layers as draft

Train the model to emit tokens at any intermediate layer (early exit). Target model continues from the layer after the draft exit, so draft compute is **reused** (not wasted). Max speedup: 2.16×.

### 5. Tree attention — verify multiple draft candidates at once

Instead of one draft sequence, build a **tree** of candidates. Tree attention lets the target verify all branches in one forward pass using a sparse, hierarchical mask where each token attends only to its ancestor nodes.

Example: four candidate completions of "machine ..."
```
machine → learning → algorithm → is
       → learning → system → design
       → translation → models → are
       → translation → system → design
```
Tree has 11 nodes, verified in one forward pass. Accept/reject by tree traversal using SpecJudge at each node.

### 6. Medusa — multiple draft heads inside one model

Add K extra LM heads to the final hidden layer:
- Head 1: predicts token t+2
- Head 2: predicts token t+3
- ...
- Head K: predicts token t+K+1

All heads produce candidates in parallel → combine into s₁×s₂×...×sK candidate sequences → verify via tree attention. Single model deployment; no separate draft weights.
