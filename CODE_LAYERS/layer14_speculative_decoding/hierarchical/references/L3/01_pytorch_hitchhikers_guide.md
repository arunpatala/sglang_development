## Summary

Speculative decoding makes educated guesses about future tokens within a single forward pass, with a verification mechanism guaranteeing correctness. The output is identical to vanilla decoding. IBM deployed this in a production environment with thousands of daily users and observed:

- **2× speedup** on Llama2 13B, Llama3 8B, Granite 7B
- **3× speedup** on Granite 20B Code

---

## The inference challenge: two key problems

Standard speculative decoding has two engineering challenges to solve before it's efficient:

1. **KV cache duplication:** Each speculator head would naively need its own KV cache, which kills throughput at scale.
2. **Verification correctness:** Attention masks must be modified to correctly verify the N+1'th token without deviating from the original model's output.

### IBM's solutions

**Problem 1 — KV cache:** Modified vLLM's paged attention kernel to enable efficient shared KV cache maintenance across all heads. No separate KV per head.

**Problem 2 — Attention masks:** Modified attention masks to enable verification of all K speculative tokens in a single forward pass while maintaining causal correctness.

```
Standard generation:
  - each step: [prompt + prev_tokens] → one new token
  - K tokens = K forward passes

Speculative generation:
  - draft: [state_Z] → [t+1, t+2, t+3] (K new tokens in parallel via heads)
  - verify: [prompt + K draft tokens] → K probabilities in one forward pass
  - K accepted tokens = 1 forward pass
```

## The MLP Speculator architecture

Rather than using a separate smaller model as the draft, IBM's approach attaches **multiple speculative heads** to the LLM itself. The architecture is hierarchical — each head stage predicts one token and feeds it to the next head stage.

```
Base model forward pass → state vector Z
                          ↓
Head 1: Z → predict t+1
             ↓
Head 2: Z + embed(t+1) → predict t+2
                           ↓
Head 3: Z + embed(t+2) → predict t+3
```

This is explicitly inspired by Medusa but modified to be **hierarchical** (each head conditions on the previous head's prediction) rather than independent. The state Z from the base model flows through all heads.

### Key configuration choices

| Task type | Number of heads | Observed speedup |
|-----------|----------------|------------------|
| Language models (7B–13B) | 3–4 heads | ~2× |
| Code models (20B) | 6–8 heads | ~3× |

More heads = more compute overhead + more lookahead. Code models benefit from more heads because code is more repetitive and predictable. Language models see diminishing returns beyond 3–4 heads.

## Speculator training: two-phase approach

Training a speculator is not the same as training a language model. The two phases address different problems:

### Phase 1: Input alignment (long context, small batches)

- Input: standard text data with long sequences (4K tokens)
- Loss: standard causal LM cross-entropy per head
- Goal: teach each head to map from Z to valid next tokens

### Phase 2: Output alignment / model distillation (short context, large batches)

- Input: outputs generated by the base model itself (256 tokens)
- Loss: force heads to match what the frozen base model actually outputs
- Goal: align the "personality" of the heads to the base model's decisions

**Ratio:** 5:2 steps of Phase 1 to Phase 2 (found empirically to work well).

The key insight of Phase 2: the speculator is trained on what the base model *would have chosen*, not just what appears in training corpora. This is critical because the speculator needs to match the base model's distribution, not a general language distribution.

**Per-head loss curves:**
- Head 1 (t+1): converges fastest — easiest prediction task
- Head 2 (t+2): converges slower — depends on head 1's prediction
- Head 3 (t+3): highest loss throughout — hardest; depends on heads 1 and 2

Tracking per-head loss separately prevents easy predictions at t+1 from masking degradation at t+3.

## Production results

IBM deployed in an internal production environment with thousands of daily users. Metrics measured: TTFT (Time to First Token) and ITL (Inter-Token Latency) across different concurrent user loads.

### Llama2 13B (chat)
- Nearly **2× faster** at all batch sizes
- Both TTFT and ITL reduced consistently

### Granite 20B Code
- Nearly **3× faster** at all batch sizes
- Larger code model benefits more because code is more predictable

### Key efficiency observation

> "We begin to observe throughput reduction beyond a batch size of 64, which happens rarely in practice."

At batch sizes > 64 the GPU shifts from memory-bound to compute-bound — at that point the parallel verification cost starts to dominate and speedups shrink. In real production (most requests are low-concurrency), this threshold is rarely reached.

## Efficiency tradeoffs: compute vs memory vs heads

Three interacting dimensions:

| Dimension | Effect of increasing heads |
|-----------|---------------------------|
| Compute | More forward pass overhead per token |
| Memory | More round-trips to HBM if heads are poor predictors |
| HBM savings | Each correct speculative token saves one full round-trip |

The tradeoff: increasing heads increases compute, but if speculator quality is high, each correct prediction saves an HBM round-trip (loading 140GB model weights). The memory savings dominate the compute cost when acceptance rates are reasonable.

**Rule of thumb:** If the speculator gets 3 tokens right in a row, you have saved 3 full HBM round-trips for the price of slightly more compute per pass.

## How this maps to Layer 14

| IBM production concept | Layer 14 code |
|----------------------|---------------|
| Modified paged attention kernel | `RadixAttention` + `KVPool` in `spec_runner.py` |
| Modified attention masks for verification | `_verify_extend()` causal mask logic |
| KV cache sharing (no duplicate KV per head) | `DraftReq` mirrors `Req`'s KV pool via `draft_kv_pool` |
| Acceptance/rejection with bonus token | `_accept_reject()` + K+1 token output |
| 3 heads → 3 draft tokens | `num_spec_tokens = 3` |
| Per-head loss tracking | Extension: `_total_proposed` per position in `lesson/07_statistics.md` |
| Phase 2 distillation | L5 exercise: "train a speculator for your Layer 14 draft model" |

The IBM speculator approach (MLP heads on target model) is an **alternative architecture** to Layer 14's two-`ModelRunner` approach. Layer 14 uses a separate small model; IBM folds the draft heads into the target model. Both share the same verification loop structure.

## Limits of this article (for book context)

- MLP Speculator architecture is more advanced than Layer 14's classic two-model draft. Read this after understanding Layer 14's `SpecRunner` design.
- Production environment is IBM TGIS, not SGLang — specific implementation details differ.
- The article doesn't show source code for the inference loop (see [fms-extras](https://github.com/foundation-model-stack/fms-extras) for that).
- Throughput reduction at batch > 64 is the key production caveat not mentioned in most L2 articles.
