# 03 — Draft KV Mirroring

For speculative decoding to work, the draft model must generate its candidate tokens autoregressively from the same context as the target model. If the target has committed tokens `[t0, t1, ..., t_k]` into its KV pool, the draft's KV state must cover exactly the same `k+1` tokens before the draft phase begins. This requirement is called KV mirroring: the draft pool's committed state always matches the target pool's committed state for the same request.

---

## The DraftReq Object

```python
# spec_runner.py — DraftReq
class DraftReq:
    """Parallel request state for the draft model."""
    def __init__(self, req_pool_idx: int):
        self.req_pool_idx:  int       = req_pool_idx
        self.slot_indices:  List[int] = []
        self.output_ids:    List[int] = []       # mirrors target req.output_ids
        self.draft_tokens:  List[int] = []       # N candidates from current spec step
```

`DraftReq` is a lightweight sibling of `Req`. It stores only the draft model's pool-management state — its `req_pool_idx` in the draft `req_to_token_pool` and its `slot_indices` in the draft `kv_pool`. The `draft_tokens` list holds the N candidate tokens generated during the draft phase of the current `spec_decode_step`; it is cleared at the start of each step.

`SpecRunner` maintains `self._req_to_dreq: Dict[int, DraftReq]` keyed by `id(req)`. When a request is prefilled, `SpecRunner.prefill(req)` creates the corresponding `DraftReq`.

---

## Prefill: Seeding Both Models

```python
# spec_runner.py — SpecRunner.prefill
def prefill(self, req: Req) -> None:
    # Step 1: target prefill
    self.target_mr.prefill_batch([req])
    # req.kv_committed_len updated, req.output_ids has first token

    # Step 2: draft prefill — same tokens, same positions
    d_req = DraftReq(req_pool_idx=self.draft_mr.req_to_token_pool.alloc())
    self.draft_mr.prefill_batch_for_dreq(req, d_req)
    # d_req.slot_indices updated, d_req mirrors target's committed state

    self._req_to_dreq[id(req)] = d_req
```

`prefill_batch_for_dreq` is a `ModelRunner` method that takes the same `req.fill_ids` and `req.kv_committed_len` as the target prefill, but runs the forward pass through the draft model and writes K/V into the draft pool. After this, the draft pool contains the same logical sequence (the prompt's KV representation through the draft model's layers) that the target pool contains through the target model's layers. The KV values are different — the draft model has different weights — but they represent the same token sequence up to the same position.

After both prefills, the first output token `req.output_ids[-1]` was sampled from the target model's logits. The draft model's `d_req.output_ids` is set to match (`d_req.output_ids = [req.output_ids[-1]]`). This ensures that when the draft phase starts, both the target and draft model consider the first generated token as part of the context.

---

## The Mirroring Invariant

At the start of every `spec_decode_step`, the mirroring invariant holds:

- The target model's KV pool for request `req` contains tokens `req.input_ids + req.output_ids[:-1]` (all confirmed tokens except the current input token, which has not yet been stored).
- The draft model's KV pool for `d_req` contains the same logical sequence through the draft model's layers.
- `len(d_req.output_ids) == len(req.output_ids)`.

During the draft phase, the draft model generates N tokens autoregressively. These tokens are appended to `d_req.draft_tokens` and also temporarily to `d_req.output_ids` (since each draft decode step uses `output_ids[-1]` as the current input token). At the end of the verify phase, the accepted tokens are committed to both the target pool and the draft pool. Rejected positions are rewound (section 06 explains the KV rewind).

After each `spec_decode_step`, the invariant is restored: both pools cover exactly the accepted prefix plus the confirmed last token, and `len(d_req.output_ids) == len(req.output_ids)`.

---

## What Happens to Draft KV During Acceptance

When `accept_len` tokens are accepted (positions `d1, ..., d_{accept_len}`), the draft model's KV pool already has those tokens written (they were written during the draft phase). No additional draft pool write is needed for the accepted tokens — they are already there.

The bonus token (the target's prediction at the rejection site) was not generated by the draft model. Its K/V must be computed and written by the draft model. This requires one additional draft decode step after the verify phase:

```python
# spec_runner.py — after accept/reject
bonus_tok = accept_result.bonus_token
d_req.output_ids.append(bonus_tok)
req.output_ids.append(bonus_tok)

# Rewind draft KV to accept_len, then write bonus_tok
self._rewind_draft_kv(d_req, accept_len)
self.draft_mr.decode_step_single(d_req, bonus_tok)  # writes bonus_tok's K/V into draft pool
```

After this, the draft pool covers `prompt + all_accepted + bonus_tok` — matching the target pool exactly — and the invariant holds for the next `spec_decode_step`.
