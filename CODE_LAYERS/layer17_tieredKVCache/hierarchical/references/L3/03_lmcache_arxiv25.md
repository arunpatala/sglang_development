# LMCache: An Efficient KV Cache Layer for Enterprise-Scale LLM Inference

**Source:** https://arxiv.org/abs/2510.09665

**Authors:** Yuhan Liu et al. (Columbia University, UChicago, etc.)
**GitHub:** https://github.com/LMCache/LMCache
**Level:** L3 — Academic paper; production alternative to HiCache
**Why here:** LMCache is the **alternative hierarchical cache** offered in SGLang via `--enable-lmcache`. Unlike HiCache (which is embedded in the SGLang memory manager), LMCache is a **middleware layer** that sits between the inference engine and storage. Understanding the difference explains why SGLang ships both and when to choose each. Layer 17's `06_configuration.md` mentions LMCache as the alternative; this paper provides the full technical comparison.

## Problem

KV cache has traditionally been stored in GPU memory. However, it is increasingly necessary to **move KV caches outside GPU devices** to:
1. Enable cache reuse across different queries
2. Enable cache reuse across different inference engines
3. Support workloads that exceed GPU VRAM capacity

Real-world usage confirms: over time, total KV cache stored by users grows **far beyond GPU memory capacity**.

---

## LMCache Solution

LMCache is described as "the first and most efficient open-source KV caching solution" that:
- Extracts and stores KV caches generated by vLLM and SGLang **outside GPU memory**
- Shares KV caches **across engines and queries**
- Supports **both** cache offloading (prefix reuse) and PD disaggregation (cross-GPU transfer)

### Three Core Contributions

#### 1. Highly Optimized KV Cache Data Movement
- **Batched data movement operations** — process multiple KV pages together
- **Compute and I/O pipelining** — overlap GPU computation with KV transfers
- **Layer-wise streaming** — dedicated CUDA streams per layer, avoiding HOL blocking

#### 2. Modular KV Cache Connector
- Decouples LMCache from the rapid evolution of inference engines
- The connector component implements the engine-side API (vLLM Connector API, SGLang plugin API)
- New engine versions require only connector updates, not LMCache core changes

#### 3. First-class Control API
- Flexible cache orchestration across GPU, CPU, storage, and network
- APIs for: capacity management, eviction policy control, cross-engine sharing, prefix invalidation

### Supported Backends

LMCache supports: CPU DRAM, local disk (NVMe), Redis, S3-compatible object storage, remote network storage.

## Results

Combining LMCache with vLLM achieves:
- Up to **15× throughput improvement** on multi-round QA
- Up to **15× throughput improvement** on document analysis

---

## Enterprise Insights (Production Data)

Large-scale deployment in enterprise settings revealed:

1. **Fetching KV from remote storage measurably benefits prefill delay** (confirmed at scale)
2. **Context truncation halves prefix cache hit ratio** — widely-applied industrial technique that discards prefix cache because the token sequence changes when context is truncated
3. **Cross-engine sharing is the key differentiator** — sharing KV across multiple inference instances on different GPUs dramatically raises effective cache hit rate

## LMCache vs HiCache: Architecture Comparison

| Aspect | LMCache | HiCache |
|---|---|---|
| **Position in stack** | Middleware layer (external to engine) | Embedded in SGLang memory manager |
| **Engine coupling** | Loose — connector API abstracts the engine | Tight — direct access to `RadixCache`, `HiRadixCache`, `HiCacheController` |
| **KV structure** | Layer-wise streaming (dedicated CUDA stream per layer) | Page-first contiguous blocks |
| **Backend support** | CPU, disk, Redis, S3, remote network | Mooncake, 3FS, NIXL, AIBrix, file, dynamic |
| **Cross-engine sharing** | First-class feature | Via L3 shared backends (Mooncake, 3FS, NIXL) |
| **PD disaggregation** | Supported (cross-GPU KV transfer) | Via Mooncake TransferEngine (separate subsystem) |
| **Cache key** | Token hash prefix | RadixTree token prefix |
| **Positional encoding** | No change | No change |
| **Prefetch policies** | Configurable pipelining | `best_effort`, `wait_complete`, `timeout` |
| **Write policies** | Configurable | `write_back`, `write_through`, `write_through_selective` |
| **SGLang flag** | `--enable-lmcache` | `--enable-hierarchical-cache` |

### When to Use HiCache vs LMCache in SGLang

**Prefer HiCache when:**
- You need tight integration with SGLang's RadixCache for maximum performance
- You are using Mooncake, 3FS, or NIXL as your storage backend
- You want fine-grained control over prefetch and write-back policies

**Prefer LMCache when:**
- You need cross-engine KV sharing across vLLM + SGLang mixed deployments
- Your storage is Redis or S3 (native LMCache backends)
- You want a middleware approach that can be updated independently of the inference engine

## SGLang Integration Point

```bash
# Enable LMCache in SGLang (alternative to HiCache)
python3 -m sglang.launch_server \
  --model-path /model --tp 8 \
  --enable-lmcache

# Enable HiCache in SGLang (default recommended approach)
python3 -m sglang.launch_server \
  --model-path /model --tp 8 \
  --enable-hierarchical-cache --hicache-ratio 2
```

SGLang's LMCache integration: `python/sglang/srt/mem_cache/storage/lmcache/`

## Key Takeaways for Layer 17

- LMCache and HiCache solve the same problem with different architecture tradeoffs.
- **LMCache = middleware** (flexible, multi-engine); **HiCache = embedded** (tighter, more optimized for SGLang).
- LMCache's **cross-engine sharing** is its strongest differentiator — HiCache achieves this via shared L3 backends.
- Context truncation is a major practical enemy of prefix caching — worth checking in production deployments.
- SGLang ships both; choose based on your storage backend and multi-engine requirements.
