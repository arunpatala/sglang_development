# Resources — technical teaching & technical writing

Curated links for **teaching** technical material and **writing** technical books or long-form guides. Pairs with `GENERAL_GUIDELINES.md` and `SPECIFIC_GUIDELINES.md` in this folder.

---

## How people learn (workshops, courses, lessons)

- [**Teaching Tech Together**](https://teachtogether.tech/en/) — Greg Wilson. Free online book on how people learn, lesson design, delivery, exercises, and building teaching communities. Source: [github.com/gvwilson/teachtogether.tech](https://github.com/gvwilson/teachtogether.tech).

---

## Evidence-based design (cognitive load, multimedia)

- [**Mayer’s principles for multimedia learning (summary)**](https://instructionaldesign.io/toolkit/mayer/) — Compact overview of Richard Mayer’s research-backed principles (coherence, signaling, segmenting, redundancy, contiguity, personalization, etc.).

- [**e-Learning and the Science of Instruction**](https://www.wiley.com/en-us/e+Learning+and+the+Science+of+Instruction%3A+Proven+Guidelines+for+Consumers+and+Designers+of+Multimedia+Learning%2C+5th+Edition-p-9781394177387) — Richard E. Mayer and Ruth C. Clark (Wiley). Deeper reference on guidelines for multimedia instructional design.

---

## Structuring technical content (modes: learn / do / lookup / understand)

- [**Diátaxis**](https://diataxis.fr/) — Daniele Procida’s framework: **tutorial** (learning-oriented), **how-to guide** (task-oriented), **reference** (information-oriented), **explanation** (understanding-oriented). Useful for splitting a book + companion site so readers know which mode they are in.

- [**Tutorials vs how-to guides (Diátaxis)**](https://diataxis.fr/tutorials-how-to/) — Short, precise distinction between “at study” vs “at work” content.

---

## Writing technical books (process, audience, craft)

- [**Ten simple rules for writing a technical book**](https://pmc.ncbi.nlm.nih.gov/articles/PMC10414569/) — Open-access article-style checklist (PLOS Comput Biol “Ten Simple Rules” series).

- [**How to Write a Technical Book**](https://third-bit.com/2022/06/19/how-to-write-a-technical-book/) — Greg Wilson. Practical notes from an experienced author/educator.

- [**So You Want to Write a Technical Book**](https://terathon.com/blog/technical-book.html) — Eric Lengyel. Author perspective on scope, motivation, and expectations.

- [**From Idea to O’Reilly Bestseller: My Complete Guide to Writing and Publishing a Technical Book**](https://dredyson.com/from-idea-to-oreilly-bestseller-my-complete-guide-to-writing-and-publishing-a-technical-book/) — Dre Dyson. Proposal, positioning, publishing path (filter for what matches your goals).

## Writing for developers (prose, docs mindset, clarity)

- [**Writing for Developers**](https://www.oreilly.com/library/view/writing-for-developers/9781633436282/) — Piotr Sarna and Cynthia Dunlop (O’Reilly). Craft of technical writing aimed at developer audiences.

- [**Technical Writing for Software Developers**](https://www.oreilly.com/library/view/technical-writing-for/9781835080405/) — Chris Chinchilla (O’Reilly). Documentation and communication for software contexts.

## Using this list with this repo

- Use **Diátaxis** to decide whether a `CODE_LAYERS` lesson is primarily **explanation** (mental model), **tutorial** (guided path), or **reference** (anchors into code)—and split or signpost when it tries to be all three at once.

- Use **Teaching Tech Together** + **Mayer** summaries when designing **exercises, checkpoints, and segment length** so code-heavy chapters stay learnable.

- Use the **technical book** links when you need **proposal, scope, iteration, and audience** discipline separate from “correctness of the engine.”

If you want a single addition later, a short “**Plain language / style manuals**” subsection (e.g. federal plain language guidelines, Google developer documentation style) can sit alongside the O’Reilly titles—say the word and it can be appended.
